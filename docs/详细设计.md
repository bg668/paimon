# 基于 TypeScript 参考实现的 Agent Python 重写详细设计

## 一、文档目的

本文档基于 [概要设计.md](/Users/biguncle/project/uu-work/docs/概要设计.md) 展开，用于指导在 `src/paimonsdk` 目录下实现一个严格对齐 `_archive/refs` 目录中 TypeScript 语义的 Python Agent SDK。本文档不再讨论“是否应该实现这些能力”，而是明确给出工程落地所需的文件目录规划、模块职责划分、核心接口设计以及关键功能逻辑伪代码。

本次详细设计严格服从当前实现范围，只覆盖以下三类目标：其一，Agent LOOP 状态外部可感知，且可在特定节点取消，并在取消后基于现有上下文继续执行；其二，外部应用可以在不修改 LOOP 的前提下扩展工具、技能、上下文管理、系统提示词和计划任务能力；其三，只适配 OpenAI 风格的 `chat.completions` stream 与 non-stream 接口。

## 二、实现原则

实现时必须坚持“SDK 内核稳定、外部应用能力外挂”的原则。也就是说，主循环、状态机、事件系统和工具执行链路属于 Runtime Kernel 的固定能力；技能装载、系统提示词拼装、上下文压缩、任务调度等属于外部应用，应通过工具、hook、队列和配置接入，而不是改写内核。

实现时还必须坚持“行为对齐优先于代码简化”。只要某种简化会改变以下任一结果，就不能采用：消息追加顺序、事件触发顺序、工具结果落地顺序、取消后的收敛形态、continue 的边界条件、以及一次运行何时才算真正 idle。

## 四、总体模块设计

### 4.1 `src/paimonsdk/runtime/models.py`

该模块定义运行时内部统一消息模型和工具模型，是整个 Runtime Kernel 的基础类型层。应定义的核心对象包括：`TextContent`、`ImageContent`、`ThinkingContent`、`ToolCallContent`、`UserMessage`、`AssistantMessage`、`ToolResultMessage`、`AgentToolResult`、`AgentContext`、`AgentTool`、`ThinkingLevel`、`ToolExecutionMode`。

其中 `AssistantMessage` 必须保留 `stop_reason`、`error_message`、`usage`、`provider`、`model`、`api`、`timestamp` 等字段，因为取消、错误和最终收敛行为都依赖这些字段。`ToolResultMessage` 必须保留 `tool_call_id`、`tool_name`、`content`、`details` 和 `is_error`。`AgentTool` 应定义为 `Protocol` 或抽象基类，要求工具对象至少有 `name`、`label`、可选 `prepare_arguments` 和 `async execute(...)`。

### 4.2 `src/paimonsdk/runtime/events.py`

该模块定义事件结构和监听器接口。事件类型需严格覆盖：`agent_start`、`agent_end`、`turn_start`、`turn_end`、`message_start`、`message_update`、`message_end`、`tool_execution_start`、`tool_execution_update`、`tool_execution_end`。

事件对象建议采用 dataclass，而不是任意字典。原因是事件不仅供外部订阅使用，也会参与内部状态归约和测试断言。监听器建议统一接受两个参数，即事件对象和取消信号，以保持与参考实现一致的可中止监听器语义。

### 4.3 `src/paimonsdk/runtime/state.py`

该模块负责运行时状态对象。建议拆为 `MutableAgentState` 与 `AgentStateView` 两层。前者用于内部更新，后者用于对外只读暴露。状态至少包括：`system_prompt`、`model`、`thinking_level`、`tools`、`messages`、`is_streaming`、`streaming_message`、`pending_tool_calls`、`error_message`。

这里的设计重点是避免把内部可变列表直接暴露给外部。对外读取状态时，应返回列表副本或不可变视图，以保证 transcript 只能通过运行时流程推进，而不能被调用方直接改乱。

### 4.4 `src/paimonsdk/runtime/queues.py`

该模块封装 steering 队列与 follow-up 队列的共性能力。建议实现 `PendingMessageQueue`，支持 `enqueue()`、`drain()`、`clear()`、`has_items()`。`drain()` 需要支持 `all` 和 `one-at-a-time` 两种模式，这不是优化项，而是行为语义的一部分。

### 4.5 `src/paimonsdk/runtime/run_control.py`

该模块负责运行时控制对象，包括活动运行句柄和取消信号。建议实现 `CancelToken` 和 `RunHandle`。`CancelToken` 可基于 `asyncio.Event` 构建，并提供 `cancel()`、`is_cancelled()`、`raise_if_cancelled()`、`wait_cancelled()`。`RunHandle` 至少需要保存取消信号、idle future 和当前运行标识。

之所以单独拆这个模块，是为了让模型流、工具执行和监听器都能使用同一种取消机制，而不是在各处混用 `task.cancel()` 或自定义布尔标记。

### 4.6 `src/paimonsdk/runtime/config.py`

该模块定义长期配置和每轮运行配置。建议至少提供 `AgentOptions`、`AgentLoopConfig`、`BeforeToolCallContext`、`BeforeToolCallResult`、`AfterToolCallContext`、`AfterToolCallResult`。

`AgentOptions` 只保留 kernel 级长期配置：session 初始化、模型身份、工具集、上下文转换、tool hooks、队列模式、`thinking_level`、`metadata`。

`AgentLoopConfig` 只保留单轮运行必需项：模型、`stream_fn`、`convert_to_llm`、`transform_context`、消息供应器、tool hooks、工具执行模式、`thinking_level` 和 `metadata`。

provider 请求参数不再放在 runtime config 中，而是下沉到 adapter 专属配置，例如 OpenAI 适配器自己的 `request_config`。

### 4.7 `src/paimonsdk/runtime/errors.py`

该模块集中定义运行时异常。建议至少定义：`AgentRuntimeError`、`AgentAlreadyRunningError`、`InvalidContinuationError`、`ListenerOutsideRunError`、`ToolPreparationError`、`OpenAIAdapterError`。尽管许多错误最终会转换为协议消息，但内部仍需明确异常类型，便于定位与测试。

### 4.8 `src/paimonsdk/runtime/stream_handler.py`

该模块负责把模型响应统一处理为 assistant final message。它需要同时支持 stream 与 non-stream。对于 stream，应维护 partial assistant message，并在 chunk 增量中持续更新；对于 non-stream，应直接构造 final assistant message，但对主循环暴露的结果接口保持一致。

该模块的核心函数建议为 `stream_assistant_response(context, config, emit, cancel_token)`。此函数内部负责：每轮调用前的上下文转换、OpenAI 消息映射、模型调用、partial/final 收敛和 `message_start/update/end` 事件发射。

### 4.9 `src/paimonsdk/runtime/tool_executor.py`

该模块负责 assistant message 中 tool call 的全链路处理。应暴露总入口 `execute_tool_calls()`，并内部拆分为：工具提取、工具准备、顺序执行、并行执行、执行中更新事件、后处理和 tool result 落地。

这是 strict alignment 的关键模块之一，尤其要保证并行模式不是“谁先完成谁先落地”，而是“准备按顺序、执行可并发、最终落地仍按 assistant 中原始 tool call 顺序”。

### 4.10 `src/paimonsdk/runtime/loop.py`

该模块实现 `run_agent_loop()`、`run_agent_loop_continue()` 和共享的 `run_loop()`。它只关心上下文推进与事件发射，不直接改动 `Agent` 自身状态。它通过 emit 回调把事件交还给 `agent.py` 做状态归约。

### 4.11 `src/paimonsdk/runtime/agent.py`

该模块提供对外 `Agent` 对象，是 Runtime Kernel 的总入口。它需要负责：管理运行状态、创建上下文快照、组装 loop config、启动和结束一次运行、归约状态、对外提供事件订阅、提供 `prompt()`、`continue_()`、`abort()`、`wait_for_idle()`、`steer()`、`follow_up()`、`reset()` 等公开方法。

### 4.12 `src/paimonsdk/adapters/openai_chatcompletions.py`

该模块是本次唯一需要实现的模型适配层。它负责将运行时上下文映射为 OpenAI 风格 `chat.completions` 请求，将 non-stream 响应转换为统一 final assistant message，将 stream chunk 转换为统一事件流。它不负责 transcript 状态管理，也不负责工具后处理。

### 4.13 外部应用接入边界

`paimonsdk` 作为 SDK，不应内置应用目录。外部应用如需实现工具注册、技能装载、上下文压缩、系统提示拼装或计划任务创建，应在 `paimonsdk` 之外实现自己的模块，并通过 SDK 暴露的工具列表、配置对象、hook 和消息队列进行接入。

SDK 对外只承诺扩展点和运行语义，不承诺内置这些应用能力的具体实现。

## 五、核心接口设计

### 5.1 Agent 对外接口

`Agent` 对外应提供以下方法。

`prompt(input_value, images=None)` 用于新增用户提示并启动运行。

`continue_()` 用于基于当前 transcript 继续运行。如果最后一条消息是 assistant，则先尝试消费 steering，再尝试消费 follow-up，仍为空才报错。

`abort()` 用于中止当前运行。中止后不清空 transcript，只改变当前活动运行的取消信号。

`wait_for_idle()` 用于等待当前运行真正结束。这里的“真正结束”不仅包括主循环结束，还包括所有事件监听器执行完成。

`steer(message)` 用于向 steering 队列追加消息，使其在当前运行的下一轮模型调用前生效。

`follow_up(message)` 用于向 follow-up 队列追加消息，使其在当前运行本来准备结束时继续推进。

`subscribe(listener)` 用于订阅事件，返回取消订阅函数。

`reset()` 用于清空 transcript、错误状态和队列，但不应修改模型、系统提示词和工具配置。

### 5.2 Tool 接口

工具对象建议满足以下约定：名称唯一；可选参数预处理；参数校验由运行时统一执行；执行方法必须是异步的，并接受取消信号和可选的中间更新回调。

工具返回结构应统一为：

```python
AgentToolResult(
    content=[...],
    details=...,
)
```

工具执行失败时，应通过抛异常交由运行时统一包装为错误 tool result，而不是由工具自己伪造 message 结构。

### 5.3 OpenAI 适配层接口

适配层建议对运行时提供两个统一入口：

`create_message(model, context, options)` 用于 non-stream 请求，直接返回 final assistant message。

`stream_message(model, context, options)` 用于 stream 请求，返回异步事件流对象，并支持 `await result()` 获取 final assistant message。

运行时内部可以统一走流式接口；对于 non-stream，可由适配层封装成一个只发出 final 的简化事件流，从而让 `stream_handler.py` 无需区分两套主路径。

## 六、关键功能逻辑设计

### 6.1 运行生命周期

一次运行开始时，Agent 必须创建 `RunHandle`，设置 `is_streaming=True`，清空 `streaming_message` 和 `error_message`。若执行中出现内部异常，则应由外层包装逻辑构造一条失败 assistant message，并发出 `agent_end`，确保运行以协议方式结束。

运行真正结束时，需要满足以下条件：主循环已经退出；最后一个 `agent_end` 事件已经归约完成；所有监听器已按注册顺序执行完成；`wait_for_idle()` 对应的 future 被 resolve。只有此时，才可将 `is_streaming` 设为假并清理活动运行句柄。

### 6.2 主循环

主循环必须完全保留双层结构。其含义不是“为了写起来方便”，而是为了精确表达 steering 与 follow-up 的不同时机。

内层循环处理“当前这轮仍可继续推进”的情况，即存在 pending 消息或上一轮 assistant 含有 tool call。外层循环只处理“本来准备结束，但 follow-up 让它继续”的情况。

### 6.3 取消语义

取消信号一旦置位，不应粗暴丢弃当前 transcript，而应让模型流或工具执行在下一个安全边界收敛。模型流若能正常产出带 `aborted` 的 final assistant message，则直接使用；若不能，则由外层故障包装补一条失败 assistant message。工具执行若观察到取消信号，可抛异常，由运行时转换为错误或中止结果。

取消并不自动触发新的运行。继续执行需要显式由应用层再次调用 `prompt()` 或注入新消息。

### 6.4 外部应用扩展

工具扩展通过 `Agent.state.tools` 或构造参数注入。

技能加载通过外部应用拼装 `system_prompt`、实现 `transform_context` 注入或调用 `steer()` 注入。

上下文管理通过外部应用实现 `transform_context(messages, cancel_token)` 完成。

系统提示词修改通过外部应用变更 Agent 当前 `system_prompt` 完成。

计划任务创建应由外部应用作为工具或 follow-up 生成策略实现，不进入 LOOP 内部逻辑。

### 7.1 `Agent.prompt()`

```python
async def prompt(self, input_value, images=None):
    if self._active_run is not None:
        raise AgentAlreadyRunningError()

    prompt_messages = self._normalize_prompt_input(input_value, images)
    await self._run_prompt_messages(prompt_messages)
```

### 7.2 `Agent.continue_()`

```python
async def continue_(self):
    if self._active_run is not None:
        raise AgentAlreadyRunningError()

    last_message = self._state.messages[-1] if self._state.messages else None
    if last_message is None:
        raise InvalidContinuationError("no messages to continue from")

    if last_message.role == "assistant":
        steering = self._steering_queue.drain()
        if steering:
            await self._run_prompt_messages(steering, skip_initial_steering_poll=True)
            return

        followups = self._followup_queue.drain()
        if followups:
            await self._run_prompt_messages(followups, skip_initial_steering_poll=False)
            return

        raise InvalidContinuationError("cannot continue from assistant")

    await self._run_continuation()
```

### 7.3 运行包装

```python
async def _run_with_lifecycle(self, executor):
    if self._active_run is not None:
        raise AgentAlreadyRunningError()

    cancel_token = CancelToken()
    idle_future = asyncio.get_running_loop().create_future()
    self._active_run = RunHandle(cancel_token=cancel_token, idle_future=idle_future)

    self._state.is_streaming = True
    self._state.streaming_message = None
    self._state.error_message = None

    try:
        await executor(cancel_token)
    except Exception as exc:
        await self._handle_run_failure(exc, cancel_token.is_cancelled())
    finally:
        self._finish_run()

def _finish_run(self):
    self._state.is_streaming = False
    self._state.streaming_message = None
    self._state.pending_tool_calls = set()
    self._active_run.idle_future.set_result(None)
    self._active_run = None
```

### 7.4 `run_loop()`

```python
async def run_loop(current_context, new_messages, config, emit, cancel_token):
    first_turn = True
    pending_messages = await config.get_steering_messages() or []

    while True:
        has_more_tool_calls = True

        while has_more_tool_calls or pending_messages:
            if not first_turn:
                await emit(TurnStartEvent())
            else:
                first_turn = False

            for message in pending_messages:
                await emit(MessageStartEvent(message))
                await emit(MessageEndEvent(message))
                current_context.messages.append(message)
                new_messages.append(message)
            pending_messages = []

            assistant_message = await stream_assistant_response(
                context=current_context,
                config=config,
                emit=emit,
                cancel_token=cancel_token,
            )
            new_messages.append(assistant_message)

            if assistant_message.stop_reason in ("error", "aborted"):
                await emit(TurnEndEvent(message=assistant_message, tool_results=[]))
                await emit(AgentEndEvent(messages=new_messages))
                return

            tool_calls = [item for item in assistant_message.content if item.type == "toolCall"]
            has_more_tool_calls = len(tool_calls) > 0

            tool_results = []
            if has_more_tool_calls:
                tool_results = await execute_tool_calls(
                    current_context=current_context,
                    assistant_message=assistant_message,
                    config=config,
                    emit=emit,
                    cancel_token=cancel_token,
                )
                for result in tool_results:
                    current_context.messages.append(result)
                    new_messages.append(result)

            await emit(TurnEndEvent(message=assistant_message, tool_results=tool_results))
            pending_messages = await config.get_steering_messages() or []

        followups = await config.get_followup_messages() or []
        if followups:
            pending_messages = followups
            continue

        break

    await emit(AgentEndEvent(messages=new_messages))
```

### 7.5 `stream_assistant_response()`

```python
async def stream_assistant_response(context, config, emit, cancel_token):
    messages = context.messages
    if config.transform_context is not None:
        messages = await config.transform_context(messages, cancel_token)

    llm_messages = await config.convert_to_llm(messages)
    llm_context = build_openai_context(
        system_prompt=context.system_prompt,
        messages=llm_messages,
        tools=context.tools,
    )

    stream = await config.stream_fn(config.model, llm_context, config, cancel_token)

    partial_message = None
    added_partial = False

    async for event in stream:
        if event.type == "start":
            partial_message = event.partial
            context.messages.append(partial_message)
            added_partial = True
            await emit(MessageStartEvent(copy_message(partial_message)))
            continue

        if event.type in STREAM_UPDATE_EVENTS:
            partial_message = event.partial
            context.messages[-1] = partial_message
            await emit(MessageUpdateEvent(copy_message(partial_message), event))
            continue

        if event.type in ("done", "error"):
            final_message = await stream.result()
            if added_partial:
                context.messages[-1] = final_message
            else:
                context.messages.append(final_message)
                await emit(MessageStartEvent(copy_message(final_message)))
            await emit(MessageEndEvent(final_message))
            return final_message

    final_message = await stream.result()
    if added_partial:
        context.messages[-1] = final_message
    else:
        context.messages.append(final_message)
        await emit(MessageStartEvent(copy_message(final_message)))
    await emit(MessageEndEvent(final_message))
    return final_message
```

### 7.6 工具准备

```python
async def prepare_tool_call(current_context, assistant_message, tool_call, config, cancel_token):
    tool = find_tool(current_context.tools, tool_call.name)
    if tool is None:
        return ImmediateToolOutcome(
            result=create_error_tool_result(f"Tool {tool_call.name} not found"),
            is_error=True,
        )

    try:
        prepared_call = maybe_prepare_arguments(tool, tool_call)
        validated_args = validate_tool_arguments(tool, prepared_call)

        if config.before_tool_call is not None:
            before_result = await config.before_tool_call(
                assistant_message=assistant_message,
                tool_call=tool_call,
                args=validated_args,
                context=current_context,
                cancel_token=cancel_token,
            )
            if before_result and before_result.block:
                return ImmediateToolOutcome(
                    result=create_error_tool_result(before_result.reason or "Tool execution was blocked"),
                    is_error=True,
                )

        return PreparedToolCall(tool_call=tool_call, tool=tool, args=validated_args)
    except Exception as exc:
        return ImmediateToolOutcome(result=create_error_tool_result(str(exc)), is_error=True)
```

### 7.7 顺序执行

```python
async def execute_tool_calls_sequential(current_context, assistant_message, tool_calls, config, emit, cancel_token):
    results = []
    for tool_call in tool_calls:
        await emit(ToolExecutionStartEvent(tool_call.id, tool_call.name, tool_call.arguments))
        preparation = await prepare_tool_call(current_context, assistant_message, tool_call, config, cancel_token)

        if preparation.kind == "immediate":
            tool_result_message = await emit_tool_call_outcome(
                tool_call, preparation.result, preparation.is_error, emit
            )
            results.append(tool_result_message)
            continue

        executed = await execute_prepared_tool_call(preparation, emit, cancel_token)
        tool_result_message = await finalize_executed_tool_call(
            current_context, assistant_message, preparation, executed, config, emit, cancel_token
        )
        results.append(tool_result_message)

    return results
```

### 7.8 并行执行

```python
async def execute_tool_calls_parallel(current_context, assistant_message, tool_calls, config, emit, cancel_token):
    results = []
    runnable = []

    for tool_call in tool_calls:
        await emit(ToolExecutionStartEvent(tool_call.id, tool_call.name, tool_call.arguments))
        preparation = await prepare_tool_call(current_context, assistant_message, tool_call, config, cancel_token)
        if preparation.kind == "immediate":
            tool_result_message = await emit_tool_call_outcome(
                tool_call, preparation.result, preparation.is_error, emit
            )
            results.append(tool_result_message)
        else:
            runnable.append(preparation)

    tasks = [asyncio.create_task(execute_prepared_tool_call(item, emit, cancel_token)) for item in runnable]

    for prepared, task in zip(runnable, tasks):
        executed = await task
        tool_result_message = await finalize_executed_tool_call(
            current_context, assistant_message, prepared, executed, config, emit, cancel_token
        )
        results.append(tool_result_message)

    return results
```

### 7.10 事件归约

```python
async def process_event(self, event):
    if event.type == "message_start":
        self._state.streaming_message = event.message
    elif event.type == "message_update":
        self._state.streaming_message = event.message
    elif event.type == "message_end":
        self._state.streaming_message = None
        self._state.messages.append(event.message)
    elif event.type == "tool_execution_start":
        pending = set(self._state.pending_tool_calls)
        pending.add(event.tool_call_id)
        self._state.pending_tool_calls = pending
    elif event.type == "tool_execution_end":
        pending = set(self._state.pending_tool_calls)
        pending.discard(event.tool_call_id)
        self._state.pending_tool_calls = pending
    elif event.type == "turn_end":
        if getattr(event.message, "error_message", None):
            self._state.error_message = event.message.error_message
    elif event.type == "agent_end":
        self._state.streaming_message = None

    if self._active_run is None:
        raise ListenerOutsideRunError()

    for listener in self._listeners:
        await ensure_awaitable(listener(event, self._active_run.cancel_token))
```

## 八、外部应用接入方式

外部应用若要扩展工具，只需构造符合 `AgentTool` 协议的对象并加入工具列表。

外部应用若要加载技能，应自行生成系统提示词，或通过 `transform_context` 注入技能上下文，而不是改 `loop.py`。

外部应用若要做上下文压缩，应实现自己的 `transform_context()` 并绑定到 Agent 配置。

外部应用若要创建计划任务，应将“计划任务的创建动作”实现为工具，或在某个 turn 结束后通过 `follow_up()` 注入下一条消息驱动后续处理。

因此，运行时内核对外部应用的承诺应是：只要外部应用遵守消息、工具和 hook 协议，就可以不断扩展能力，而无需修改 LOOP 本身。

## 十、实施顺序建议

第一步先实现 `models.py`、`events.py`、`state.py`、`queues.py`、`run_control.py` 和 `errors.py`，把基础类型和状态控制打稳。

第二步实现 `openai_chatcompletions.py` 与 `stream_handler.py`，把 stream 和 non-stream 的统一协议打通。

第三步实现 `tool_executor.py` 与 `loop.py`，完成主循环闭环。

第四步实现 `agent.py`，把生命周期、状态归约、事件订阅和外部 API 串起来。

第五步由外部应用工程按需接入自己的工具、技能、上下文管理和计划任务模块，并基于 SDK 编写集成测试。

## 十一、结论

本详细设计给出的不是一个“大而全”的 Agent 平台方案，而是一套严格服从当前实现范围、同时又为应用层扩展留足边界的 Python Runtime 设计。只要后续实现严格遵守这里定义的目录规划、模块边界和关键伪代码，就可以在 `paimonsdk` 下构建一个与 `_archive/refs` 中 TypeScript 参考实现语义对齐、且适合后续业务扩展的 Agent LOOP 内核。

## 十二、P0-1 AgentSession / Checkpoint 详细补充

### 12.1 新增数据结构

`src/paimonsdk/runtime/session.py` 新增以下可序列化对象：

`ToolReference`：保存工具的稳定引用信息，只包含 `name`、`label`、`description`、`input_schema`。它用于 checkpoint 检查与调试展示，不保存可执行函数本体。

`QueueSnapshot`：保存 steering / follow-up 队列模式和当前排队消息。恢复后队列语义必须与导出前一致。

`StableBoundary`：描述最近一次可恢复边界，至少包含 `kind`、`captured_at`、`message_count`、`event_seq`、`last_run_id`、`last_turn_index`。`P0-1` 中 `kind` 只允许 `initialized` 和 `idle` 两类可恢复状态。

`RecordedEvent`：保存 replay 所需的最小事件日志，至少包含 `event_id`、`run_id`、`seq`、`timestamp`、`turn_index`、`type`、`payload`。这里的 payload 只保留重放关键过程需要的稳定字段，不要求与运行时内部事件对象逐字段同构。

`AgentSession`：保存完整可恢复 session，包括 transcript、system prompt、model、thinking level、tool references、两个队列快照、metadata、最近稳定边界和事件日志。

`Checkpoint`：是 `AgentSession` 在某个稳定边界上的导出包装，带 `checkpoint_id`、`created_at` 和边界镜像，用于显式表达“这是一份可导入恢复的快照”。

### 12.2 AgentSession 与运行态的对齐原则

运行中的 `Agent` 仍然维护内存态 `MutableAgentState`。`AgentSession` 不直接替代它，而是作为稳定导出协议。

对齐规则如下：

1. transcript、system prompt、model、thinking level 直接来自当前 Agent 内存态。
2. `tools` 本体保留在运行态；导出时转换为 `tool_refs`。
3. steering / follow-up 队列通过 `QueueSnapshot` 导出和导入。
4. `metadata` 和 `session_id` 作为 session 级字段持久化。
5. `is_streaming`、`streaming_message`、`pending_tool_calls` 这类瞬时字段不进入 checkpoint；它们只用于判断当前是否允许导出。

### 12.3 导出 / 导入 / Resume 流程

`export_session()`：

1. 校验当前不在运行中，且没有未收敛的 streaming / tool execution 状态。
2. 基于当前内存态构造 `AgentSession`。
3. 复制最近稳定边界和事件日志。
4. 返回可继续 `to_dict()` 的纯数据对象。

`export_checkpoint()`：

1. 先执行 `export_session()`。
2. 将 session 包装成 `Checkpoint`。
3. 复制 session 中的 `stable_boundary` 到 checkpoint 顶层，便于导入时做显式一致性校验。

`from_session(session, options)`：

1. 用 session 中的 transcript、system prompt、model、thinking level、队列和 metadata 初始化 Agent。
2. 用 `options` 提供非序列化运行期依赖，例如 `stream_fn`、hooks、实际 tool 对象。
3. session 的状态字段优先于 `options` 中的同名状态字段；`options` 只覆盖运行期能力。

`from_checkpoint(checkpoint, options)`：

1. 校验 checkpoint / session schema version 一致。
2. 校验 `stable_boundary.kind` 属于可恢复集合。
3. 校验 `message_count`、`event_seq` 等边界字段与 session 主体一致。
4. 校验通过后按 `from_session()` 恢复 Agent。

Resume 主路径不引入新的特殊 loop。恢复后的 Agent 与普通 Agent 一致：如果 transcript 末尾允许 `continue_()`，则走 `continue_()`；否则应用层继续调用 `prompt()` 发起下一轮。也就是说，resume 恢复的是“稳定运行上下文”，不是某个被暂停的底层生成器。

### 12.4 Replay 流程

`P0-1` 的 replay 基于 `RecordedEvent` 列表完成，不依赖数据库或日志后端。

推荐提供 `replay_events(event_log, run_id=None)` 或等价接口：

1. 若指定 `run_id`，按 `run_id` 过滤单轮运行事件。
2. 若未指定，则默认回放最近一轮带 `run_id` 的事件。
3. 返回按 `seq` 排序的事件列表，由 CLI / 调试器 / 测试自行渲染关键过程。

这里的 replay 目标是复现“一轮运行发生了什么”，而不是重建底层 provider 返回的每个 token。

### 12.5 API 草案

建议对外新增以下接口：

```python
class Agent:
    @classmethod
    def from_session(cls, session: AgentSession, options: AgentOptions | None = None) -> "Agent": ...

    @classmethod
    def from_checkpoint(cls, checkpoint: Checkpoint, options: AgentOptions | None = None) -> "Agent": ...

    def export_session(self) -> AgentSession: ...

    def export_checkpoint(self) -> Checkpoint: ...

    def replay_run(self, run_id: str | None = None) -> list[RecordedEvent]: ...
```

建议新增错误类型：

`SessionExportError`：当前不在稳定边界，不能导出 session / checkpoint。

`CheckpointImportError`：checkpoint 结构、schema version 或稳定边界校验失败，不能恢复。

### 12.6 测试补充

`P0-1` 需要新增以下测试：

`session export`：验证 transcript、tool refs、队列快照、stable boundary、event log 都能导出并 roundtrip。

`checkpoint import`：验证从 checkpoint 恢复后可继续执行，且 `continue_()` / `prompt()` 语义不变。

`resume`：验证从 user 结尾 transcript 的 checkpoint 恢复后可以继续跑主路径。

`replay`：验证事件日志可按最近 run 或指定 `run_id` 回放。

`boundary error`：验证运行中导出报错、非法稳定边界导入报错。

## 十三、P0-2 事件流协议稳定化详细补充

### 13.1 事件 Envelope 结构

`src/paimonsdk/runtime/events.py` 中所有运行时事件都需要继承统一 envelope，至少包括：

```python
@dataclass(slots=True, kw_only=True)
class EventEnvelope:
    event_id: str | None = None
    run_id: str | None = None
    turn_id: str | None = None
    seq: int = 0
    timestamp: int = 0
```

兼容策略是“字段新增、语义补齐、类型名不变”。也就是说，`AgentStartEvent`、`MessageStartEvent`、`ToolExecutionEndEvent` 等现有类型保留，但都带上同一套 envelope 字段。

### 13.2 关联字段补充

除 envelope 外，还要补足 turn / message / tool 的稳定关联字段：

`MessageStartEvent`、`MessageUpdateEvent`、`MessageEndEvent` 增加 `message_id`。

`ToolExecutionStartEvent`、`ToolExecutionUpdateEvent`、`ToolExecutionEndEvent` 增加 `assistant_message_id`，继续复用已有 `tool_call_id`。

`TurnEndEvent` 增加 `message_id`，指向本 turn 的 assistant message。

这样 replay 或外部 UI 可以只依赖事件协议，就把“哪个 tool call 属于哪个 assistant message、哪个 assistant message 属于哪个 turn”串起来。

### 13.3 发射链路

`loop.py`、`stream_handler.py`、`tool_executor.py` 继续发出原有事件对象；`agent.py` 在真正对外发射前统一补齐 envelope。

建议流程如下：

1. 低层模块只负责发出语义事件，不强行感知 `run_id / seq`。
2. `Agent._process_event()` 进入时先执行 `prepare_event(event)`。
3. `prepare_event(event)` 负责：
   - 从 `RunHandle.run_id` 补 `run_id`
   - 在 `turn_start` 时生成并记录新的 `turn_id`
   - 为 turn 范围内后续 message / tool / turn_end 事件补 `turn_id`
   - 生成 `event_id`
   - 分配单调递增的 `seq`
   - 写入 `timestamp`
   - 为 message 事件补 `message_id`
   - 为 tool 事件补 `assistant_message_id`
4. 补齐后的事件再进入状态归约、监听器分发和 replay 日志记录。

### 13.4 顺序与归约约束

`seq` 必须等于监听器实际观察到的顺序。也就是说，哪怕内部存在并发工具执行，对外仍以“最终发射顺序”编号。

状态归约使用补齐后的事件，而不是原始裸事件。这样 replay 日志与监听器看到的是同一份协议。

记录到 `RecordedEvent` 中的最小字段应至少包括：

```python
RecordedEvent(
    event_id=event.event_id,
    run_id=event.run_id,
    turn_id=event.turn_id,
    seq=event.seq,
    timestamp=event.timestamp,
    type=event.type,
    payload=...,
)
```

其中 payload 至少保留：

`message` 事件的 `message_id` 与序列化消息体。

`tool_execution_*` 事件的 `assistant_message_id`、`tool_call_id`、`tool_name` 及必要结果字段。

`turn_end` 事件的 `message_id` 和 tool result 列表。

### 13.5 API 与测试补充

需要新增或更新的测试重点如下：

`event protocol`：验证监听器看到的所有事件都带 `event_id / run_id / seq / timestamp`，且 turn 范围事件带 `turn_id`。

`message correlation`：验证 assistant 的 `message_start / update / end` 使用同一个 `message_id`。

`tool correlation`：验证 `tool_execution_*` 的 `assistant_message_id` 指向触发本轮工具调用的 assistant message，且多个 tool call 在同一 turn 共享相同 `turn_id`。

`replay minimum fields`：验证 `replay_run()` 返回的日志中保留 envelope 和 replay 所需的最小 payload 字段。
